Как читать отчёт

Что такое «aigw-drm-agent» в одном абзаце

Цифровой привратник / консьерж для клиентов банка. Задача агента — быстро понять, куда направить обращение: начало коммуникации, запрос обратного звонка, запрос обратного чата, вызов к свободному оператору. Модель GigaChat-2-Pro используется только как классификатор — выбирает один из пяти кодов действия. Клиенту уходят только заранее заданные шаблонные сообщения, свободного текста от модели клиент не видит.

Почему много «Нет» в чек-листе — это не катастрофа

Основной риск обычного AI-ассистента — свободная генерация текста клиенту. Здесь этот риск закрыт архитектурно: выход модели ограничен enum, клиенту отдаются только шаблоны. Поэтому формально «нет» санитизации / цензора / guardrails в runtime — это реальный пробел по Методике, но не критический потому, что вред от такого агента ограничен сверху узостью его сценария и закрытым набором действий. Доработки из колонки «Рекомендуемое действие» в обоих таблицах закрывают эти пункты и одновременно снижают оценки по группе «Безопасность данных и систем» Методики.

Почему для Методики много «Информационно»

Часть пунктов Методики — не про код, а про процесс. Критичность AI-агента считает Калькулятор критичности, степень значимости модели (A/B/C/D/E) валидирует УМР, КТН и BCP — ДТН, правовая оценка — Правовой департамент, финальный риск-рейтинг — УОР по заключениям всех профильных подразделений. Эти пункты невозможно «закрыть» только кодом, поэтому по ним стоит статус «Информационно» с указанием ответственной стороны.

Чем Таблица 3 отличается от Таблицы 1 и Таблицы 2

Таблица 1 — общий чек-лист КБ AI-агентов и МАС (фиксированные строки 58–68). Таблица 2 — Методика интегральной оценки рисков (ч. 1, ред. 16.02), ракурс «группа риска». Таблица 3 — это проектные критерии из родительской задачи DRMMMB-19: 6 подзадач (HealthCheck, Retry, StopEvent, трейс, операционный UID, предотвращение зацикливаний), которые уже заведены в Jira как обязательные доработки именно для этого КЭ. Здесь P0/P1 — приоритет реализации, а не severity. Таблицы пересекаются в нескольких местах (например, контроль токенов / rate limit из Таблицы 1 близок к Retry/StopEvent из Таблицы 3), но смотрят на агент с разных сторон и закрываются разными артефактами.

Мини-план доработок до встречи с УКИИ

Критично из чек-листа: (1) подключить Sanitizer в runtime; (2) подключить цензор / базовый детектор промпт-атак; (3) добавить rate limit и лимит попыток; (4) добавить контроль токенов (max_tokens, бюджет, метрики); (5) ввести confirm-step перед вызовом IDSS; (6) пересмотреть logRequests/logResponses и маскирование PII в логах. Желательно: снизить temperature, подключить AuditWriter.logEvent, задать maxLength для operator_input.

Критично из DRMMMB-61..66 (P0): (a) Retry с backoff/jitter для GigaChat и IDSS (DRMMMB-62); (b) обработка StopEvent / 429 / 503 от GigaPlatform с управляемым отказом (DRMMMB-63). P1 — настоящий HealthCheck с контрольной корзиной, структурированный trace c rollback flag, operation_uid / parent_operation_uid и устранение разрыва корреляции в CommunicationBackRequest, TTL/deadline на LLM/IDSS и enforcement retry_number.
